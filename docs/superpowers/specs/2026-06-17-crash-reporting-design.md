# Crash + Error Capture (Firebase Crashlytics) â€” Design Spec

**Date:** 2026-06-17
**Brick:** `bricks/riverpod_simple_architecture`
**Builds on:** sub-projects #1 (configurable brick) + #2 (settings) â€” already on `master`
**Status:** Approved design, pending implementation plan

## Context & Goal

Give every app generated from the brick **app-wide crash + uncaught-error capture** via Firebase
Crashlytics â€” behind a **no-op-default, pluggable interface** so apps compile and run with **zero
Firebase**, and a **setup script** that activates Crashlytics per app when wanted.

### Decisions locked during brainstorming
- **Scope: crash + error capture ONLY.** No analytics events, no screen-view tracking, no consent
  toggle this round (those are separate, later sub-projects).
- **No-op default + scripted opt-in.** Firebase config (`firebase_options.dart`) is per-app and
  generated by the interactive `flutterfire configure`, so it cannot be pre-baked. The brick ships a
  no-op reporter; a script wires Firebase when the developer runs it.
- **Marker-based script wiring** (over fuzzy patching) for deterministic edits.
- **Errors route to BOTH talker (always) AND the crash reporter** (no-op or Firebase).

### Non-goals (out of scope)
- Firebase Analytics events / `setUserId` / `setUserProperty` / screen tracking.
- Analytics consent UI.
- iOS Crashlytics dSYM upload automation (documented note only; Android Gradle is patched).
- Fully non-interactive Firebase project creation (impossible â€” needs the developer's account).

## Approach

**Script with marker-based auto-wiring** (chosen over "mechanical-only + manual Dart steps" and
"docs only"). `tool/setup_firebase.sh` runs the one unavoidable interactive step
(`flutterfire configure`), then deterministically adds deps, writes the Firebase impl, swaps two
**marker comments** in the brick's Dart files, and patches Android Gradle. Markers make the edits
reliable.

### `CrashReporter` interface (surface)
```dart
abstract class CrashReporter {
  Future<void> recordError(Object error, StackTrace? stack,
      {bool fatal = false, String? reason});
  Future<void> recordFlutterError(FlutterErrorDetails details, {bool fatal = false});
  Future<void> log(String message);
  Future<void> setCustomKey(String key, Object value);
}
```
`NoopCrashReporter` implements every method as a no-op. The Firebase impl forwards to
`FirebaseCrashlytics.instance`.

## Data flow

Uncaught error â†’ `FlutterError.onError` / `PlatformDispatcher.instance.onError`
â†’ `talker.handle(error, stack)` (always â€” existing logging)
â†’ `crashReporter.recordError(...)` / `recordFlutterError(...)` (no-op default, Firebase after setup).
A manual `ref.read(crashReporterPod).recordError(e, s)` is available for caught errors.

## Error handling

`wireCrashHandlers` wraps reporter calls so an exception inside the reporter is swallowed (logged to
talker) and never propagates â€” crash reporting must not itself crash the app. The no-op path keeps
full talker logging for apps without Firebase.

## Script behaviour (`tool/setup_firebase.sh`)

1. Preconditions: ensure `flutterfire` CLI present (else print install hint and exit non-zero).
2. `flutterfire configure` â€” interactive; generates `lib/firebase_options.dart` and registers apps.
3. `flutter pub add firebase_core firebase_crashlytics`.
4. Copy `tool/templates/firebase_crash_reporter.dart` â†’ `lib/shared/observability/firebase_crash_reporter.dart`.
5. Marker swap A: in `crash_reporter_pod.dart`, replace the no-op default with `FirebaseCrashReporter()` (+ import).
6. Marker swap B: in `future_initializer.dart`, replace `// FIREBASE_INIT` with `await Firebase.initializeApp(options: DefaultFirebaseOptions.currentPlatform);` (+ imports).
7. Patch Android Gradle: add `com.google.gms.google-services` + `com.google.firebase.crashlytics` to `android/settings.gradle.kts` (plugins, `apply false`) and apply them in `android/app/build.gradle.kts`. No-op-safe (warn if already present / not found).
8. Idempotent where feasible (re-running does not duplicate plugins/markers).

## Testing & verification

- **Harness-verified (no-op path):**
  - `crash_reporter_test` â€” `NoopCrashReporter` methods complete without effect.
  - `error_handlers_test` â€” `wireCrashHandlers(spy)` then trigger a `FlutterError` (e.g. `FlutterError.reportError(...)`) â†’ the spy's `recordFlutterError` is called; a reporter that throws does not crash the handler.
  - Brick generates green (analyze 0 issues, all tests pass) in BOTH responsive states.
- **Script mechanical coverage:** the Gradle-plugin patch logic is exercised on a sample gradle string (a small standalone check) so the deterministic part is tested.
- **Honest limitation:** the **Firebase-active** path (real `flutterfire configure` + Crashlytics
  reporting) cannot be harness-verified â€” it needs a real Firebase project and account. It is
  documented and manually verified once.

## Dependencies

- No new **runtime** deps in the default brick (no-op path adds nothing).
- The script adds `firebase_core` + `firebase_crashlytics` at the latest stable versions at setup
  time (pinned into the app's pubspec by `flutter pub add`).

## Success criteria

- A freshly generated app compiles, runs, and logs uncaught errors to talker with **no Firebase** â€”
  green in both responsive states.
- `wireCrashHandlers` routes `FlutterError` + async errors to the active `CrashReporter`; a throwing
  reporter never crashes the app (tested).
- Running `tool/setup_firebase.sh` (after the interactive `flutterfire configure`) leaves the app
  reporting crashes to Firebase Crashlytics, with the two markers swapped and Android Gradle patched.
