# Settings + About Page (Sub-project #2) ÔÇö Design Spec

**Date:** 2026-06-17
**Brick:** `bricks/riverpod_simple_architecture`
**Builds on:** Sub-project #1 (configurable brick ÔÇö provides `AppConfig`)
**Status:** Approved design, pending implementation plan

## Context & Goal

Give every app generated from the brick a proper **Settings / About** screen ÔÇö the
canonical home for theme, language, and app info. It consumes the per-app identity
already centralised in `AppConfig` (from #1): `appTitle`, `supportEmail`, `privacyUrl`.

### Decisions locked during brainstorming
- **Onboarding is deferred** ÔÇö not part of this sub-project (it will be its own piece later).
- **"Branding"** is already satisfied by `AppConfig`; this sub-project consumes it.
- **About items:** app name + version, Contact support (mailto), Privacy policy link.
  (A dedicated "author/made-by" row was explicitly **not** requested and is out of scope.)

### Non-goals (out of scope)
- First-run onboarding / intro flow.
- Rate-us / share (those belong to sub-project #3).
- Notification or analytics toggles (sub-projects #3/#4 add their own settings rows later).
- Restructuring the counter demo beyond adding the settings entry point.

## Approach

**Routed Settings page** (`/settings` via auto_route), reached from the app bar via a
reusable gear button ÔÇö chosen over a modal bottom sheet (cramped for About) and inline-on-home
(doesn't scale). This matches the existing feature-first structure and gives a real settings
surface every future app keeps.

## Data flow & wiring

- **Theme:** reuse `ThemeSegmentedBtn` (bound to `themecontrollerProvider`) inside an Appearance section.
- **Language:** reuse `AppLocalePopUp` (bound to `translationsPod`) as the trailing of a Language row.
- **Version:** `packageInfoPod` (FutureProvider) rendered with the existing `easyWhen` extension; shows `${info.version}+${info.buildNumber}`.
- **About actions:** Contact row Ôćĺ `launchEmail(AppConfig.supportEmail)`; Privacy row Ôćĺ `launchWebUrl(AppConfig.privacyUrl)`.
- **Navigation:** `SettingsButton` pushes `SettingsRoute`; the counter app bar gains this button alongside the existing `AppLocalePopUp`.
- **Labels:** localized via slang (`t.settings.*`), consistent with the rest of the app. The base ships English + Spanish values; apps localize further.

## Error handling

- `launcher_helper` returns `false` on `canLaunchUrl`-fail or launch exception (never throws to the UI).
- On a `false` result, the Settings page surfaces a flash error via the existing `GlobalHelper` snackbar pattern ÔÇö no silent failures.
- **Empty-config guard:** when `AppConfig.supportEmail` (or `privacyUrl`) is empty, that row is **not rendered**, so apps that leave those blank show no dead links.

## Testing

Widget tests via the existing `test/helpers/pump_app.dart` + `spot`/`mocktail`:
- Settings page renders the Appearance, Language, and About sections.
- Version renders once `packageInfoPod` is overridden with a fake `PackageInfo`.
- Contact / Privacy rows render only when their `AppConfig` values are non-empty; tapping invokes the launcher helper (mocked).
- `SettingsButton` is present in the counter app bar and navigates to the settings route.

End-to-end verification reuses sub-project #1's harness: `tool/verify_brick.sh <name> <responsive>` Ôćĺ `mason make` (with hooks: slang + build_runner) Ôćĺ `flutter analyze` (0 issues) Ôćĺ `flutter test` (all pass), run for both `responsive=false` and `responsive=true`.

## Dependencies

- `package_info_plus` (latest stable) ÔÇö runtime version/build info.
- `url_launcher` (latest stable) ÔÇö mailto + web URL launching.
- Both pinned in `__brick__/pubspec.yaml` per the pinning convention from #1.

## Success criteria

- A generated app shows a gear button in the home app bar that opens a Settings page with:
  Appearance (theme selector), Language (locale selector), and About (app name + version,
  Contact support, Privacy policy).
- Contact/Privacy rows open the mail composer / browser, and are hidden when their config is empty.
- Labels are localized (slang); the project analyzes clean and passes tests in both responsive states.
- The counter demo is unchanged except for the added settings button.
