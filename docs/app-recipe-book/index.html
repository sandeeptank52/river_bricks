00What this is

The Recipe Book is documentation plus one committed Claude skill (flutter-app-planner) that turns the question “what kind of app is this?” into an auditable list of ingredients and a written base product plan. It is meant to be used before any code is written.

  • Plan-only. The planner emits two files — base-product-plan.md and ingredient-checklist.md — and stops. No scaffolding, no code generation, no brick.yaml manifest.
  • Portable. Everything lives in-repo (docs/app-recipe-book/ + .claude/skills/), so any developer who clones the brick gets it for free — no personal setup.
  • Deterministic. The same interview answers always yield the same ingredient set.
  • Readiness bar = “Solid default.” Every generated plan ships a production baseline; heavier modules are added only when the app calls for them.

01How it works

STEP 1

Interview

Ask the 13 questions, one cluster at a time, to characterise the app.

STEP 2

Decide

Apply the decision matrix: 9 must-haves + app-type bundle + per-answer signals − conflicts.

STEP 3

Emit

Write base-product-plan.md + ingredient-checklist.md into the new project.

STEP 4

Stop

Hand back to the developer. The app is built separately from the plan.

The planner reads ingredient-catalog.md (definitions), decision-matrix.md (the engine), production-foundation.md (the baseline), and the recipe docs. The interview script is embedded in SKILL.md so it works even with no docs open.

Conflict & prerequisite rules

  • Digital goods → RevenueCat, never Cashfree. A third-party gateway for digital content violates store policy and risks removal.
  • Local-only backend → drop cloud-sync (and usually authentication); keep offline-support & local-database.
  • ads selected → must implement the ATT prompt (iOS 14.5+) and Google UMP consent.
  • ai-integration → always pair with rate-limiting (AI bills spike fast). Its avoid-unless-needed tier means “don’t add gratuitously” — AI is exactly when it is needed.
  • referral-system → always add deep-links (attribution must survive a fresh install).

05App-type recipes

Each recipe = the 9 must-haves + this bundle + whatever the interview signals add.

leanest

Utility / tools

Adds: nothing required
Backend: often local-only
Monetize: ads or one-time unlock
  • 9 must-haves are enough to ship a solid tool
  • Engagement/monetization layers are per-signal
read-heavy

Content / feed

Adds: deep-links remote-config performance-monitoring
Backend: Firebase or a CMS/API
  • Links drive discoverability; config tunes the feed without releases
  • Image-heavy lists need perf monitoring
offline-first

Productivity / notes

Adds: local-database offline-support onboarding
Backend: local-only → optional cloud-sync
  • Losing a user’s writing is unforgivable
  • Cloud-sync + auth added when multi-device matters
heaviest

Subscription / SaaS / AI

Adds: authentication payments-revenuecat remote-config onboarding
Backend: Firebase / Supabase / custom
  • Entitlements follow the user; store IAP for digital goods
  • AI → ai-integration + rate-limiting

06Backend recipes

Backend is chosen per app. Firebase is the lean default (it matches the brick’s Crashlytics work).

Firebase · default

  • Auth, Firestore, Functions, FCM, Remote Config, Crashlytics
  • Realtime via Firestore streams; fast start; generous free tier
  • Best for: quick start + realtime

Supabase

  • Postgres + Row-Level Security + Realtime + Storage + Edge Functions
  • Best for: relational/SQL data, SQL-native teams

Custom REST / GraphQL

  • Typed client (dio + retrofit / ferry); JWT auth
  • Best for: an existing server or full control

Local-only

  • hive_ce (brick); no accounts, no sync
  • Clean upgrade path to a backend later

07Monetization

Two payment paths are fundamentally different — do not conflate them.

RevenueCat store IAP

For: digital goods & subscriptions inside the app
Pkg: purchases_flutter
  • Required by App Store / Play for digital content
  • Hosts receipts/entitlements — no server needed for basic flows
  • Always implement “restore purchases”

Cashfree direct gateway

For: real-world goods/services, India market
Pkg: flutter_cashfree_pg_sdk + server
  • Server required: create order → SDK collects → verify webhook → fulfill
  • Never confirm fulfillment on the client; KYC + PCI care
  • Not for digital goods (store rejection)

Ads AdMob

For: free utility/content apps
Pkg: google_mobile_ads
  • ATT (iOS) + GDPR/UMP consent before personalized ads
  • Use test ad unit IDs in development
Conflict rule: if you sell digital subscriptions or in-app unlocks, use RevenueCat (store IAP) — not Cashfree. Cashfree is for physical/real-world transactions.

08Analytics · Security · Testing · Release

Analytics events

  • snake_case, verb_noun; default sink firebase_analytics
  • Core: app_open, screen_view, sign_up/login, error_shown, settings_changed
  • Per type: tool_run · content_view/search_performed · item_created/sync_completed · paywall_view/purchase_completed/ai_request

Security checklist

  • Secrets: no keys in source/history (--dart-define, server-held)
  • Storage: sensitive data in secure storage, not prefs
  • Transport: HTTPS only; cert pinning only if the threat model needs it
  • Payments/PII: never confirm payment on the client; minimise PII
  • Web: flutter_secure_storage is not secure on web (localStorage) — keep tokens off the client
  • Account deletion: in-app deletion is required when you offer accounts (Apple 5.1.1(v) — hard rejection)

Testing strategy

  • Notifier unit tests via testNotifier (brick helper)
  • Widget tests for key screens; flutter analyze clean in CI
  • Add per ingredient: payments restore, offline conflict, AI cost-guard
  • Don’t over-test generated code (*.g.dart, router.gr.dart)

Release checklist

  • Metadata: name, description, keywords, screenshots, icon, version
  • Privacy: policy URL live; App Store / Play data labels; ATT/UMP
  • CI/CD gates: analyze + test green; apk + ios builds pass; Crashlytics symbols uploaded

10Use & extend

How to use

  • Run /flutter-app-planner (committed skill) — or paste claude-starter-prompt.md into any chat.
  • Answer the interview; the planner writes base-product-plan.md + ingredient-checklist.md into your new project.
  • Build the app from the plan using the brick + the checked ingredients.
  • See example-plan.md for a complete worked output (a subscription AI app).

How to extend

  • New ingredient → add a classified entry to ingredient-catalog.md, then wire it into decision-matrix.md and any relevant recipe.
  • New app type → add a recipe section + a base-bundle row in decision-matrix.md.
